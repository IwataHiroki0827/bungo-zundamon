# ドメイン調査 F002（継続コンテンツ拡充）

## 1. Tier判定

調査Tierは**大**とする。本フィーチャーは外部公開済みの静的Webアプリへ文学作品と合成音声を継続追加するものであり、著作権、青空文庫収録ファイルの利用条件、VOICEVOXソフトウェア利用規約、ずんだもん音源・キャラクター利用ガイドライン、GitHub Pagesの容量制限が同時に関係する。誤選定した作品または権利条件を満たさない音声・画像を公開した場合の影響が大きく、音声容量と全件編集レビューの工数もバッチごとに累積するため、KB検索だけでは不十分と判断した。

## 2. 調査対象と方法

- 対象プロジェクトのF001成果物、青空文庫公式の公開作家一覧・作家ページ・図書カード・配布テキストを確認した。
- 青空文庫公式配布テキストをファイルへ保存せずメモリ上で読み、`「」`候補数、本文量、長台詞の概算を比較した。
- VOICEVOX公式規約、東北ずん子・ずんだもんプロジェクト公式の音源・キャラクター規約を確認した。
- 候補選定は別評価役が反証し、長文、括弧形式、作品内容、レビュー工数、音声容量の弱点を再評価した。
- 法的断定ではなく、追加時と公開直前の公式書誌・規約再確認を受入条件とする。

## 3. 用語集

| 用語 | 本文書での意味 |
|---|---|
| バッチ | 1作者と代表3作品を、権利確認から試験まで一まとまりで追加する単位 |
| 候補 | 青空文庫本文領域の全角かぎ括弧`「」`から機械抽出した、編集確認前の文字列 |
| 承認済み台詞 | 原典文脈を確認し、登場人物の発話またはまとまりのある内心として`approved`にした候補 |
| 編集除外 | 注意書き、手紙、判決文、作者引用、語句例、抽出artifact等を理由付き`rejected`にした候補 |
| 差分生成 | 正規化文、VOICEVOX ENGINE版、話者・スタイル、音声設定から決まる音声IDを使い、未生成・変更分だけ生成する処理 |
| 容量preflight | 音声生成前に作業ドライブ空き容量、追加見積り、公開総容量、単一ファイル容量を判定する処理 |

## 6. 権利・利用条件

### 6.1 青空文庫

- 3作品の図書カードは宮沢賢治本人を`著者`とし、翻訳者を記載していないため、日本語原著候補として扱える。
- 選定時と公開直前に、公式拡充CSVの`人物著作権フラグ=なし`、`作品著作権フラグ=なし`、`役割=著者`、公開中、新字新仮名版を再確認する。
- 青空文庫の「著作権の切れている作品」は複製・再配布・共有・実演・口述・翻案・形式変換が可能である。一方、作品名、作者、底本、入力者、校正者、加工内容等の由来情報を保持する。
- 公式ファイルは修正され得るため、取得日時と原典SHA-256を記録し、hash変更時は候補抽出・全件レビュー・音声差分をやり直す。

### 6.2 VOICEVOX・ずんだもん

- VOICEVOXで生成した音声を利用する場合、ソフトウェア規約と各音声ライブラリ規約の双方に従う。
- アプリから少し探せば分かる場所に、正確な`VOICEVOX:ずんだもん`クレジットを置く。
- 文学作品中の表現であること、作者・作品・原典への導線を台詞と結び付け、ずんだもん自身の主張と誤認させない。
- 非商用の二次創作はガイドライン範囲で利用できるが、第三者の二次創作画像には別の権利がある。新作者画像は独自制作・生成物または条件を確認した素材だけを用いる。
- 収益化、スポンサー、企業販促、キャラクター商品化を導入する場合は、現行判断を流用せず別フィーチャーで再評価する。

## 7. 容量と配信制約

- F001は59台詞、音声約26.7 MiBである。初回3作品の候補をすべて音声化する最悪寄りの概算は約57～76 MiBで、追加後の音声総量は約83～103 MiBとなる。
- 1バッチの追加音声は100 MiB以下を受入条件とする。
- 公開総容量500 MiB以上で警告し、新規追加方法や圧縮方式を再評価する。
- 公開総容量が750 MiBちょうどなら警告し、750 MiBを超える見込みなら音声生成と公開を停止する。
- GitHub Pagesの公開サイト上限1 GBへ近づく前に停止できるよう、750 MiB超をProjectFactory側の停止基準とする。Pages公開物とsource repositoryの1 GBは安全側で1,000,000,000 bytesと数え、source repositoryは750,000,000 bytes以上で警告、1,000,000,000 bytes以上で停止する。
- 単一GitオブジェクトはGitHubの100 MiB制限へ近づけず、音声は台詞単位の小さいファイルとする。
- 音声はボタン操作まで取得しない。通常閲覧では同一originの静的assetだけを使い、外部音声APIへ接続しない。

## 8. 業界慣行・当たり前機能として維持する事項

- 作者一覧、作者固有URL、作品別の台詞・出典・音声をデータ駆動で解決する。
- 原文表示と読み上げ文を分離し、読み補正の変更履歴を保持する。
- 自動抽出結果を無審査公開せず、`approved/rejected/pending`と理由を全件保持する。
- 音声は遅延読込し、自動再生せず、同時再生を1件に制限する。
- 出典・クレジット・非公式表示を全ページから1操作以内で確認できるようにする。
- 既存作者・作品・音声を回帰試験し、新規バッチによる混線・404・セキュリティ退行を防ぐ。

## 9. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 注意書きや引用を台詞として公開 | 内容品質低下、作者表現の誤提示 | 原典文脈を含む全件編集レビュー、`pending: 0`をゲート化 |
| 擬音・固有語の読み誤り | 音声品質低下 | 表示文/TTS文分離、読み補正レビュー、実音声抜取確認 |
| 作者間のroute・音声参照混線 | 誤表示、別作品音声の再生 | author/work/dialogue/audio IDの参照整合検査、作者切替時停止 |
| 音声の累積肥大化 | clone・build・Pages公開失敗 | 新規WAV、最終Pages artifact、Git object storeを別々に算定する段階判定、差分cache、作品単位のpreflight |
| 原典や規約の更新 | 権利・クレジット不整合 | 追加時と公開直前の一次情報再確認、確認日・hash証跡 |
| 文学表現をずんだもん自身の主張と誤認 | キャラクターイメージ・規約リスク | 作者・作品・文学引用である旨を台詞ごとに表示 |
| 第三者二次創作の無断利用 | 権利侵害 | 独自制作/生成または権利証跡付きallowlistに限定 |
| 複数作者対応によるXSS・外部通信退行 | 公開サイトの安全性低下 | text node描画、CSP、同一origin、依存監査、secret scan、通信0件回帰 |

## 10. ドメイン由来の要求候補

1. 単一作者・3作品のハードコードを廃止し、複数作者をcatalogから解決する。
2. 1作者・3作品を継続投入単位とし、作品単位で権利確認から受入まで完結させる。
3. 人物・作品の著作権フラグ、役割、翻訳者、公開状態を選定時と公開直前に検証する。
4. 候補全件の編集レビューと`pending: 0`を公開条件とする。
5. 固定VOICEVOX条件と音声IDで差分生成し、承認済み台詞の生成失敗を0件にする。
6. 新規WAV、最終Pages artifact、Git object storeを別々に数え、1バッチ100 MiB、公開総量500/750 MiB、Pages公開物とsource repositoryの10進1 GBを機械検査する。
7. 後続バッチもProjectFactoryの新フィーチャーとして4承認ゲートを通す。
