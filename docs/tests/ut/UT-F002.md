# 単体試験仕様書 F002（継続コンテンツ拡充）

## 試験方針

- `DD-F002.md`の`FUN-F002-001`〜`FUN-F002-040`を各1ケース以上で網羅する。
- 各ケースは正常入力に加え、仕様上の境界値・異常入力・状態不整合を同一ケース内のparameterized subcaseとして実行する。subcase IDは`UT-F002-NNN.normal`、`.boundary.<name>`、`.error.<name>`、`.recovery.<name>`とする。
- 自動化対象はVitestを基本とし、jsdom、fake timer、in-memory transport、fetch/Audio mock、Git adapter、filesystem fault injection、およびsubcase固有の一時workspace/cacheを使用する。clock、環境変数、module、DOM、Audio、network spyは`afterEach`でresetし、単独実行・逆順実行でも結果を変えない。
- 例外・検証失敗時はfail-closedとする。target rename前は既存artifactをbyte不変で維持する。rename後はdurability point前でもtargetがexpected old/new SHAのどちらかに完全一致すればその版へ収束してよく、第三者値は上書きしない。durability point後はjournalのexpected new SHAと一致する検証済み新版へ収束する。旧版/検証済み新版の完全な一方だけを残し、部分file、未所有orphan、lock、秘密情報、未公開台詞全文を残さない。
- state遷移試験は`BatchStatus`と`WorkStatus`を表駆動で全edge検査し、skip、rewind、未完了gate、古いevidence、後続work先行を拒否する。入力objectを変更せず、同じ確定済みevidenceによるrecovery/retryは同一結果になることを確認する。
- atomic試験は一時file write、file sync、可能なOSでのdirectory sync、競合再確認、rename、再open/hash検証、journal recoveryの各境界へfaultを注入する。冪等性試験は同一expected SHA、tree digest、journal ID、candidate tupleで再実行し、二重rename・二重manifest更新・二重publishが起きないことを確認する。
- security試験はpath traversal、symlink/junction/reparse point、SSRF、DNS rebinding、redirect/proxy、危険DOM、open redirect、外部origin、secret/stack/response body漏えいを含める。実network、VOICEVOX、公開repository、実Pagesは使用しない。
- 容量は整数byteで検査する。追加音声`104,857,600`は許可し`+1`で停止、Pagesは`524,288,000`以上警告、`786,432,000`ちょうど警告・`+1`停止、Pages/repositoryは`1,000,000,000`以上停止、repositoryは`750,000,000`以上警告、単一Git objectは`104,857,600`以上停止とする。

## KB-0001を反映したpath/hash契約

単体試験が生成側・消費側それぞれに都合のよいpath/hash fixtureを別々に作って不一致を隠さないよう、次の共有contract fixtureを全該当ケースで使用する。生成側関数が返した値・保存したcanonical JSONを再構成せず、そのまま消費側validator/resolverへ渡す。
