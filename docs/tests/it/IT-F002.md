# 結合試験仕様書 F002（継続コンテンツ拡充）

## 1. 試験方針

本仕様書は、F002のbatch定義、青空文庫原典、全件レビュー、VOICEVOX差分生成、作品単位受入、複数作者catalog/UI、権利・容量・セキュリティ検査、clean checkoutからのreleaseを結合した振る舞いを検証する。純粋関数内の細かな型・境界値はUTへ委ね、ITでは複数moduleと複数artifactを実際に接続し、状態、path、ID、SHA-256、byte数、candidate tupleが境界を越えて同じ意味を保つことを確認する。

- KB-0001「単体PASSでも結合で壊れるパス契約不一致」を明示反映する。生成側だけ、消費側だけのfixtureで終わらせず、生成側が保存したworkspace相対POSIX pathまたは公開root相対POSIX pathをmanifest/catalogへ記録し、その値を次stage、public tree builder、Vite Pages preview、ブラウザの`resolvePublicAsset()`まで通す。同じ流れでauthor/work/dialogue/audio/batch ID、artifact SHA-256、bytesも全段で照合する。
- 外部取得とVOICEVOX ENGINE接続は明示更新シナリオだけで使用する。通常build、work preview、release-verify、ブラウザ試験では固定fixtureと追跡済みaccepted sourceだけを使用し、networkを全拒否する。
- 各ITは固有の一時workspace、cache、staging、lock、journal、固定UTC時計、fake transport/audio adapterを使用する。終了時にadapterとbrowser contextをresetし、単独・逆順・再実行で同じ結果になることを確認する。
- 正常系では各stageの入力hash、出力hash、tool版、件数、時刻を証跡化する。異常系では理由code、停止stage、更新前後のmanifest/public/accepted-audio digestを保存し、token、環境変数、stack、任意response body、未公開台詞全文を診断へ含めない。
- 作品はmanifest順の「よだかの星」`000473`→「どんぐりと山猫」`043752`→「注文の多い料理店」`043754`で処理し、各作品の`pending: 0`、容量forecast、音声生成、actual容量、音声完全性、F001不変、content/dist preview、atomic受入を完結してから次作品へ進める。
- 共通不変条件は、transaction開始前またはtarget rename前の失敗では検証済みF001 baseline、先行accepted work、cache、追跡済みpublicをbyte不変で維持することである。rename後はjournalのexpected old/new SHAへ完全一致する版だけへ収束してよく、durability point後は検証済み新版を正とする。第三者値を上書きせず、未検証stagingまたは別candidateを公開・受入状態へ昇格しない。
