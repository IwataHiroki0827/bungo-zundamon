# 機能設計書 F002（継続コンテンツ拡充）

## 1. 設計方針

F001の完全静的・同一origin・明示再生という境界を維持し、単一作者と固定3作品に依存する実装を、承認済みbatchを統合するデータ駆動構造へ置換する。F002は宮沢賢治（人物ID `000081`）の「よだかの星」`000473`、「どんぐりと山猫」`043752`、「注文の多い料理店」`043754`をこの順に処理する。ブラウザは統合済みの`public/content/catalog.json`だけを読み、青空文庫・規約snapshot取得とVOICEVOX生成は明示CLIでのみ行う。

信頼境界は `batch入力 → 原典固定 → 抽出 → 全件レビュー → 容量判定 → 音声生成 → batch受入 → 公開tree統合` とする。各境界の入力hash・出力hash・状態をbatch manifestへ記録し、前段のhashが変わった後段artifactを流用しない。公開treeはF001と全accepted batchから毎回再構築し、active batchだけで既存treeを上書きしない。

## 2. データ・batch設計

### DES-F002-001: 複数作者catalog schema [REQ-F002-002][REQ-F002-011]

`CatalogV2`は`schemaVersion='2.0.0'`、`authors[]`、`works[]`、`audioAssets[]`、`batches[]`、全体・batch別の候補集計、`creditsRef`を持つ。`Author`は`authorId/name/originalName/slug/artwork/introducedByBatchId/identitySha256`、`Work`は`workId/authorId/title/cardLink/source/dialogues[]/batchId`、`Dialogue`は`dialogueId/workId/audioId/...`を持つ。author/work/dialogue/audio/batch IDとauthor slugは各namespaceで一意とし、`work.authorId → author`、`dialogue.workId → work`、`dialogue.audioId → audio`、各要素のbatch参照を全件検証する。同じ作者の後続batchはauthor identityとartwork hashが完全一致する場合だけ既存Authorへworkを追加し、相違時は衝突停止する。generic schemaは作者数1以上を許可し、F002受入で2作者以上を要求する。作者数・作品数に製品上限や特定ID条件を置かず、raw catalog 8 MiB、文字列32,768 Unicode scalar、深さ64と容量予算で資源消費を制限する。

### DES-F002-002: BatchManifestと工程状態 [REQ-F002-005][REQ-F002-019]

batch定義は`content/batches/{batchId}/batch.json`に保存する。`batchId/feature/status/author/workIds/workProgress/schemaVersion/inputPaths/outputPaths/stageRecords/rightsSnapshotIds/voiceConfigRef/artworkProvenanceRef`を常時必須とし、`acceptedAt/acceptedBy`はaccepted以降、`publishedAt/releaseVersion/deploymentEvidenceRef/smokeEvidenceRef`はpublishedで必須とする。F002のauthorは`000081/みやざわずんじ/宮沢賢治/miyazawa-zunji`、work IDsは上記3件の順序固定とする。各`WorkProgress`はtemplate生成時の`pending`から`extracted → reviewed → budget-approved → voiced → accepted`へ進む単方向状態、入力/出力hash、容量予測・実測証跡を持つ。manifest順で直前workが`accepted`のときだけ次workを`reviewed`以降へ進め、1作品ごとに`pending: 0 → 容量予測 → 一時音声生成 → 実測容量 → 音声完全性 → F001不変検証 → 統合preview受入`を完結する。batchの`reviewed/budget-approved/voiced/accepted`は全workの最低状態から導出し、`pending` workがある間はbatch前段状態を維持して、先行workの直列処理を妨げない。各stageは入力artifactのcanonical SHA-256、ツール版、出力artifact SHA-256、件数、RFC 3339日時を記録する。

### DES-F002-003: F001 immutable baseline [REQ-F002-001]

F001基準をrelease `v0.1.0`、commit `2733b5fd368e847a01708724511f993f5e1b2484`、release catalog SHA-256 `5125e1c788adf95d247eae6c072e2afe010937b9af78cb292effbdf31649f1c1`に固定する。`content/baselines/F001-v0.1.0.json`にはsynthetic `published` batch descriptor、芥川作者identity、license/artwork/provenance manifest hash、3作品・59台詞ごとの`workId/dialogueId/displayText/speechText/cardLink/sourceSha256/provenancePath/provenanceSha256/audioId/audioPath/audioSha256/audioBytes`とbaseline自身の生成元を記録する。clean cloneの現行`public`を明示的なsource rootとして受け取り、baseline allowlistにあるF001実体をreparse拒否・hash全件一致後だけ`F001BaselineBundle`へ昇格する。F002統合前後でこの項目集合と全F001 asset実体hashを比較し、catalog全体hashやschema表現は比較しない。不一致はF002で補正せず、変更管理を要求して公開を停止する。

### DES-F002-004: 原典・抽出・正規化・全件レビュー [REQ-F002-005][REQ-F002-006][REQ-F002-007][REQ-F002-008][REQ-F002-009][REQ-F002-010]

既存の安全な青空文庫transport、charset検査、本文DOM token化、外側`「」`抽出、決定的IDを再利用し、author/work allowlistをBatchManifestから供給する。選定時とrelease候補SHA確定後のdeploy直前に公式書誌を別々に取得し、人物ID`000081`、人物・作品著作権フラグ`なし`、役割`著者`、翻訳者なし、公開中、新字新仮名、図書カード/本文URLを`WorkRightsObservation`へ記録・比較する。deploy直前観測はrelease commitと結合し、不一致・欠落・古い観測では公開を停止する。図書カード、本文URL、底本、入力者、校正者、取得日時、raw SHA-256、ツール版、加工をwork provenanceへ保存する。レビューは各作品の全候補を`approved/rejected`へ確定し、理由、revision、reviewer、日時、規約判断を必須にしてworkごとの`pending: 0`をgateとする。表示文とspeech textを別フィールドに保持し、補正前後・理由・revision・reviewerを`SpeechRevision[]`へ残す。

## 3. 音声・公開tree・UI設計

### DES-F002-005: VOICEVOX差分cacheと完全性 [REQ-F002-012][REQ-F002-013]

音声IDはcanonical化した`{speechText, engineVersion, speakerUuid, speakerName, styleId, styleName, speedScale, pitchScale, intonationScale, volumeScale, outputSamplingRate, presetVersion}`のSHA-256とする。F002の生成前にENGINE `0.25.2`、speaker UUID `388f246b-8c41-4ac1-8e2d-5d79f3ff56d9`、speaker名`ずんだもん`、style ID `3`、style名`ノーマル`を実APIと完全一致させる。再生成省略cacheは`.cache/voice/{configHash}/{audioId}.wav`、clean checkoutで再現できるimmutable source候補は`content/batches/{batchId}/accepted-audio/{workId}/{audioId}.wav`、公開生成先は`audio/{batchId}/{audioId}.wav`を使う。同一keyは再生成0件、1台詞または設定変更は対応keyだけmissにする。生成前forecastに合格後、miss WAVをwork専用一時directoryへ上限付きで保存し、F002全workの新規unique WAV累積が100 MiBを超える前に停止する。生成後の実測容量、音声完全性、content/dist preview、F001不変がPASSした場合だけ、batch lockとjournalの下で当該workの新規unique WAVを空のimmutable work directoryへ一括atomic昇格し、path/SHA/bytesを受入証跡・manifestに固定する。先行work sourceは別directoryのため置換しない。source昇格とmanifest更新の間で停止した場合は、expected manifest/tree digestとexact hashが一致するorphanだけを冪等再開し、不一致は隔離して停止する。cacheはaccepted sourceから再構成可能な補助コピーとして扱う。失敗時は既存accepted source/cache/publicを維持する。各WAVのRIFF/WAVE構造、SHA-256、bytes、duration、config hashを検査し、approved台詞の欠損、失敗、孤立assetが1件でもあればwork/batchを`accepted`にしない。

### DES-F002-006: accepted batchの公開tree統合とpath契約 [REQ-F002-001][REQ-F002-011]

公開生成は検証済みF001 baseline bundleとF002以降の公開対象BatchManifestをbatch ID昇順で読み、author/work/dialogue/audio/provenance/license/artworkを1つのCatalogV2とtreeへ統合する。F001はbundleからだけsynthetic `published` batchとして`batches[]`へ含める。同一authorはidentity/artwork hash一致時だけmergeする。作品受入時は`work-preview` modeで、既にpublishedなbatch、active batch manifest、同batch内の先行accepted work証跡・accepted-audio、当該workのreview・音声完全性PASS済みstagingを一時統合し、現workまでの累積content treeを作る。続けて同treeをViteの一時publicDirとしてoffline buildし、`index.html`、JS、CSS、`.nojekyll`を含む完全なdist Pages previewでactual容量と参照整合を測定する。F001不変はcontent treeと最終distの双方で検査し、同じ入力・dist SHAに結合する。content previewとdist previewは公開昇格せず、work accepted後の証跡としてだけ固定する。releaseは三段階とする。まず全accepted-audio、manifest、コードをcommitして`sourceCommit`を確定する。prepare-releaseはそのexact clean sourceCommitから`releaseCandidateBatchId/feature`で既published全件＋今回のaccepted候補1件を統合し、検証済みpublicだけを更新してcommitする。続いてpublic commitを`releaseCommit`とした`ReleaseBuildContext`を確定し、exact SHAの別clean checkoutでrelease-verifyを読み取り専用実行する。release-verifyはtracked publicを変更せず、同じpublished全件＋候補1件をtracked accepted-audioから再buildしてtracked publicとbyte一致させる。他の未published batch混入を停止し、candidate tupleをゲート④、dist SHA、artifact digest、deploy、published遷移まで結合する。`sourceCommit/releaseCommit`はpublic/catalog bytesへ埋め込まず外部build metadataだけに保持し、固定点循環を作らない。生成前のF001 bundle構築と統合後のF001不変検証を別契約にし、CatalogV2への循環依存を持たせない。生成側がmanifestへ書くpathは常に公開root相対POSIX path（先頭slash、`.`、`..`、backslash、scheme、query、fragmentなし）とし、例は`audio/F002/{audioId}.wav`、`artwork/miyazawa-zundamon.png`である。消費側はVite base `/bungo-zundamon/`を`Location.origin`へ絶対化し、共通`resolvePublicAsset()`だけでURL化する。filesystem側はworkspace絶対pathに正規化し、symlink/junction/reparse pointを拒否する。prepare-releaseの公開昇格はexclusive lock、期待current-public hash、random staging、段階journal、各rename直前のidentity/hash再検査、rename後hash検証、起動時recoveryを必須とする。Windowsの`EBUSY/EPERM`はbounded retry後に旧treeを維持して停止し、検証済みbackup以外を復元しない。

### DES-F002-007: 複数作者route・一覧UI [REQ-F002-003][REQ-F002-004][REQ-F002-005][REQ-F002-020]

routeは`#/`、`#/authors/{slug}`、`#/credits`とし、構文解析後にCatalogV2の一意slug集合で解決する。未知・重複・decode不能slugはnot-foundとし外部遷移しない。トップは作者ごとに表示名、`原著者: ...`、作品数、公開台詞数、画像をカード表示する。作者ページは`authorId`で所属workだけを抽出し、作者間混線を防ぐ。宮沢カード・作者ページ・出典は`みやざわずんじ`と`宮沢賢治`を別ラベルで併記する。hash routeのため作者URLの直アクセス・再読込でもPages 404を起こさない。

### DES-F002-008: 音声再生とroute切替 [REQ-F002-014][REQ-F002-020]

Audio要素はアプリ全体で1インスタンスを維持する。button操作前は音声requestを0件とし、別台詞の前に現音声を停止、author/route変更時に`stop()`して`currentTime=0`へ戻す。404、破損、decode、play拒否は対象dialogueの状態だけを`error`へ遷移させ、日本語理由と再試行を表示する。他作者・他台詞・navigationは継続可能とする。再描画時はevent listenerを破棄し、古いbuttonへ状態を通知しない。

## 4. 権利・容量・安全設計

### DES-F002-009: 規約snapshot・権利判定・クレジット [REQ-F002-006][REQ-F002-007][REQ-F002-015]

青空文庫取扱規準、VOICEVOXソフトウェア規約、ずんだもん音源規約、キャラクター・画像条件を用途別exact canonical HTTPS URLでallowlist化する。取得はcredentials、query、fragment、任意portを禁止し、DNS全回答の公開IP検査と接続IP pin、Host/SNI/証明書検証、redirect・proxy・retry禁止、status/media type/byte/timeout上限を通す。選定時とdeploy直前の`PolicyObservation`に`policyId/url/finalUrl/status/mediaType/responseBytes/fetchedAt/contentSha256/transportVersion/versionOrLabel/reviewer/decisionSummary`を保存する。raw bytesはgitignore済み`.cache/rights/`だけに置き、public repository、Pages、Actions artifactへ含めない。commitするのはhash・取得日時・URL・版・判定要約だけとし、public buildでsnapshot混入0件を検査する。2時点のhashが異なる場合は影響範囲を`dialogue/audio/artwork/credit`で明示し、再レビュー完了まではfail-closedとする。全route footerに`VOICEVOX:ずんだもん`、非公式、広告・スポンサーなしを表示し、creditsで作者・作品・図書カード・底本・入力・校正・加工、規約URLと確認日、日本法基準・国外免責を確認できるようにする。

### DES-F002-010: 宮沢賢治作者画像provenance [REQ-F002-016]

F002画像は独自生成物または利用条件確認済み素材だけを入力allowlistへ登録する。`content/batches/F002/artwork-provenance.json`に実際のprompt・編集recipeと各hash、model/provider/version、生成時規約観測、全参照入力の由来・license・path・hash、出力`artwork/miyazawa-zundamon.png`のhash、ずんだもん画像ガイドライン判断、第三者素材・標章の目視結果、reviewer、reviewedAt、creditを記録する。hashは改ざん検知に使い、判断内容の代替にしない。第三者二次創作物、由来不明素材、再配布不可素材、allowlist外入力を拒否する。公開tree内の実体hashとmanifestが一致しない場合はbuildを停止する。

### DES-F002-011: 4区分容量preflight [REQ-F002-018]

容量は生成前`CapacityForecast`と生成・統合preview後`ActualCapacityReport`を別stage recordに保存し、次の4区分を混同せず測定する。forecastはmiss推定、現accepted tree、新規Git候補、staging/cache安全余裕、作業driveを用い、actualはF002の新規unique WAV実体、最終staging、`.git/objects`実測と未object化候補を用いる。

| 区分 | 算入範囲 | 警告 | 停止 |
|---|---|---:|---:|
| 追加音声 | 当該batchで新規生成するunique WAVの非圧縮ファイルbytes | なし | `> 104,857,600`（100 MiBちょうどは許可） |
| Pages artifact | 最終artifact内の全regular fileの非圧縮bytes | `>= 524,288,000` | `> 786,432,000`、または`>= 1,000,000,000`（750 MiBちょうどは警告のみ） |
| Git source repository | `.git/objects`実測と新規unique Git object見込の重複排除合計 | `>= 750,000,000` | `>= 1,000,000,000` |
| 単一Git object | 新規・既存各Git blob/objectのbytes | なし | `>= 104,857,600` |

生成前と各WAV write前に作業ドライブ空き容量を確認し、staging、cache、生成出力と安全余裕を満たさなければ停止する。単一WAVは残りbatch音声予算以下の上限付きstreamで一時保存し、生成後actual PASS前にcache/publicへ昇格しない。Git forecastはobject IDで既存pack/looseと重複排除し、新規unique blobはraw bytesを保守値として加算する。全値を整数bytesで計算し、閾値、算入path、重複排除hash、判定理由を証跡へ出す。

### DES-F002-012: runtime・更新工程のセキュリティ [REQ-F002-017]

閲覧時は同一originの静的assetだけを許可し、CSPは`default/script/style/img/media/connect-src 'self'`、`object/base-uri/form-action/frame-src 'none'`を維持する。`innerHTML`、inline handler、`eval`、open redirect、外部TTS、Cookie、storage、form、analyticsを使わない。catalogの表示値は`textContent`、asset pathは公開相対path検査、外部linkは用途別canonical URL検査と`noopener noreferrer`を通す。更新transportは既存のTLS/hostname/DNS pin/redirect・proxy禁止/byte上限を維持し、VOICEVOXはloopback固定とする。secret scan、依存Critical/High、CSP違反、許可外requestのいずれかでreleaseを停止する。

### DES-F002-013: UI・アクセシビリティ・ブラウザ品質 [REQ-F002-020]

semantic heading/list/link/button、可視focus、44 CSS px以上のtarget、`aria-pressed`と`aria-live`、reduced motionを維持する。390×844、844×390、1440×900で作者カード増加と長い台詞を折返し、横overflowと重なりを起こさない。キーボードでトップ→各作者→再生→停止→creditsを完結できるようにする。既存337 UT・78 browser試験をF001回帰baselineとし、F002試験を追加してChrome/Edge/iOS SafariおよびPlaywright Chromium/Firefox/WebKit/Android相当で重大退行0件をrelease条件にする。

## 5. 継続運用・異常処理

### DES-F002-014: 後続batchの再実行契約 [REQ-F002-019]

CLIを`content:batch -- --batch {featureId} --work {workId} --stage {stage|all}`へ統合し、authorと代表3作品をBatchManifestから受け取る。`--work`はmanifest記載IDだけを許可し、前のworkがacceptedでなければ後のworkを拒否する。`--stage all`は手動reviewまたは作品受入gateで正常停止し、再実行時にhash chainから再開する。各後続batchは新featureとして要求・設計・テスト仕様・releaseの4承認ゲートを必須にし、`権利確認 → 原典固定 → 抽出 → 全件レビュー → 容量forecast → 差分音声の一時生成 → actual容量 → 作品単位受入 → 統合受入`を同じschemaとコマンドで実行する。同一作者追加はauthor identity一致時だけmergeする。次回作品「雪渡り」と次作者「太宰治」は`content/batch-candidates.json`に候補としてだけ記録し、承認済みBatchManifestや公開catalogへ自動昇格しない。

### DES-F002-015: fail-closed・atomicity・診断 [REQ-F002-001][REQ-F002-009][REQ-F002-013][REQ-F002-015][REQ-F002-018]

ID/hash/path/参照/状態不整合、pending残存、規約変更未判定、F001 baseline差異、容量停止条件はwork、batchまたはbuild全体を理由code付きで停止し、既存publicを変更しない。VOICEVOXの台詞単位失敗は既存の検証済みcacheを維持するが、当該workの一時生成物を昇格せず`voiced/accepted`へ進めない。manifest更新は既存SHA-256を確認し、同directoryの一時fileへUTF-8 canonical JSONを書き、file syncを必須、directory syncを対応OSで実施後にrenameする。Windowsでdirectory sync非対応の場合はrename後の再open/hash検証とjournalで補い、能力を証跡へ記録する。複数fileの公開はexclusive lock、expected public hash、staging、journal、bounded retry、recoveryを通す。診断にtoken、環境変数、stack、任意response body、未公開台詞全文を含めない。

### DES-F002-016: CI・受入・release gate [REQ-F002-001][REQ-F002-017][REQ-F002-018][REQ-F002-020]

CIは通常buildでnetworkを禁止し、typecheck、lint、UT、統合、CatalogV2 schema、F001 baseline、全参照、容量4区分、CSP・secret・依存監査、Playwrightを実行する。checkoutは`persist-credentials:false`、Actionsは最小権限・40桁SHA pinとする。既にpublicの後続releaseではvisibilityを変更せず、ゲート④の対象F002・承認SHA・main/deploy/artifact SHAを一致させ、一回限りdeploy変数を必ず無効化する。`build`はbatchをacceptedのまま候補artifactだけ生成し、ゲート④承認、deploy成功、公開後全route smokeをrelease commitと結合した専用`publish`処理だけが`published`へ遷移させる。F001の初回release証跡は変更しない。F002は3作品を指定順に各`pending:0/音声完全/容量PASS`で受入後、全体でF001不変、作者間混線0、全route、作品権利と規約の2時点、browser証跡を確認する。いずれか未完了ならreleaseゲート④へ進めない。
