## 問題

- QT結果を同じbatch・feature・release commit・dist・artifactへ結合する共通証跡契約がなかった。
- manifestのatomic rename後に停止・競合した場合のdurability pointと回復oracleが不明確だった。
- VoiceConfig、規約transport、権利観測鮮度、disk安全余裕の境界が判定可能な値・式になっていなかった。
- exact release commitでの容量再実測、Actionsの40桁SHA pin、deploy後published manifestのatomic永続化が試験不足だった。
- Windows実filesystem回復、transport別SSRF、実Git pack/loose経路がmock中心だった。

## tm.jsonによる影響抽出

`tm.json`のF002 graphから、上記契約を担うDES-F002-002/005/006/009/011/012/015/016を起点にFUN・UT・ITを抽出した。直接修正対象はDD-F002、UT-F002、IT-F002、QT-F002であり、要求本文SRS-F002は変更しない。変更後はT-031でレビューパネルとtrace_checkを再実施する。

## 変更方針

- atomic writeはtarget rename前を旧版とし、rename後・durability point前は実体がexpected old/new SHAの完全な一方ならその版へ収束する。durability point後は検証済み新版を正とし、第三者競合を上書きしない。
- 既存実装で固定済みのVoiceConfig範囲とAozora transport上限をF002契約へ明記する。
- 観測鮮度は任意の経過時間ではなく、同じreleaseCommit・phase・実行chainへの結合で判定する。
- 空き容量は同時に存在し得る新規・backup bytes合計と安全余裕式で判定する。
- work受入reportをreleaseへ流用せず、exact clean releaseCommitで全容量区分を再実測する。
- 全QT証跡を共通candidate tupleへ結合し、F001不変は最終distとupload対象digestまで照合する。
